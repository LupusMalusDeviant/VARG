Varg (.varg) & AI-OS

Das erste Betriebssystem & die erste Programmiersprache, nativ gebaut für autonome KI-Agenten.

Warum Varg (.varg)?

Herkömmliche Sprachen wie Python oder C# wurden für deterministische Prozesse und klassische Web-/Desktop-Anwendungen gebaut. Varg ist anders. Es behandelt KI-Modelle, Kontexte und Agenten nicht als externe API-Aufrufe, sondern als native Bestandteile der Sprache.

Die Kern-Features

  • Interface First: Verträge (contract) sind das Rückgrat der Sprache.
  • C# meets Rust: Die elegante, objektorientierte Syntax von C#, gepaart mit der gnadenlosen Speichersicherheit und Performance von Rust unter der Haube.
  • Rust Transpilation Backend: Varg-Code wird nach Rust transpiliert und via cargo build zu nativen Binaries kompiliert. Nutzt Rusts gesamtes Ökosystem als Sicherheitsnetz.
  • OCAP Security: Absolute Sicherheit durch Capability-basierte Tokens. Keine Manifest-Dateien, keine Hacks.

Unterschiede zu anderen Sprachen

Feature Python / C# Varg (.varg)
KI-Datentypen Externe Bibliotheken (Strings für Prompts) Nativ (Prompt, Context, Tensor)
Concurrency Async/Await (Threads blockieren) Actor Model (Asynchrones Message-Passing)
Tool Calling Manuelles JSON-Schema Mapping Automatisches Compiler-Mapping via contract
Sicherheit OS-Level Berechtigungen (User) Capability-Tokens im Code (OCAP)

Das Ziel: Ein User-Space AI-OS

Sobald der Varg-Compiler steht, ist der nächste Schritt die Entwicklung des Betriebssystems. Das AI-OS setzt auf einem minimalen Linux-Kernel auf und übernimmt die Orchestrierung.

  • Vektorgraph-Dateisystem: Das OS nutzt Vektorgraphen (Knowledge Graph + Embeddings) als natives Speichermedium anstelle von flachen Byte-Dateien.
  • Hybrides Model-Multiplexing: Das OS fungiert als VRAM-Manager. Es multiplext große Modelle (z.B. Llama 3) intelligent für alle Agenten durch und lädt kleine Modelle parallel in die GPU.
  • Hibernation-Daemons: Agenten laufen als Daemons. Wenn der VRAM knapp wird, friert das OS sie ein (Snapshot in den Vektorgraphen) und weckt sie in Mikrosekunden bei Bedarf wieder auf.
  • Zero-Copy Memory Sharing: Agenten tauschen gewaltige Kontextfenster aus, ohne sie im VRAM zu kopieren, was Latenz und Speicherverbrauch massiv senkt.

# Architektur-Dokument: AI-OS & "Varg" (.varg)

**Mission Statement:** Entwicklung eines hochperformanten User-Space KI-Betriebssystems (auf Minimal-Linux) und einer nativen Programmiersprache für LLMs und Agenten-Systeme.
**Tech-Stack:** Varg Source (.varg) -> vargc (Rust Compiler) -> Rust Source -> cargo build -> Native Binary.

---

## 1. Syntax & Grundregeln ("Interface First")
* **C#-Inspiration:** Die Syntax orientiert sich stark an modernem C# (geschweifte Klammern `{}`, Modifier wie `public/private`).
* **Keyword `contract`:** Definiert Interfaces und Werkzeuge.
* **Typensystem:** Streng und explizit typisiert. Keine implizite Typ-Inferenz bei Signaturen.

## 2. Architektur der Akteure
* **Kern-Konstrukte:** Strikte Trennung zwischen `struct` (passiv) und `agent` (aktive KI-Prozesse/Daemons).
* **Fehlerbehandlung:** Explizit und strikt (`Result`). Keine unsichtbaren Exceptions.
* **Nebenläufigkeit (Actor Model):** Asynchrone Kommunikation über Message-Passing. Zero-Copy Memory Sharing durch das OS.

## 3. Native KI-Features
* **First-Class Types:** `Prompt`, `Context`, `Embedding` und `Tensor` sind native Datentypen.
* **Tooling-"Magie":** Vollautomatisches LLM-JSON-Schema Generierung aus `contract` Definitionen durch den Compiler.
* **Context-Management:** Automatisches Sliding-Window. OS lagert überschüssigen Kontext in Vektorgraphen aus ("Eject").

## 4. OS-Ressourcen & "Gedächtnis"
* **Vektorgraph-Dateisystem:** Vektorgraphen als natives Speichermedium für Langzeitwissen.
* **Hybrides Model-Multiplexing:** Intelligentes Queuing großer Foundation-Modelle + paralleles Laden kleiner Modelle.
* **Hardware-Zuweisung:** Dynamischer Load-Balancer (CPU, GPU, NPU) mit expliziten `@target` Overrides.

## 5. Ökosystem & Sicherheit
* **Git-First Paketverwaltung:** Dezentraler Import von Modulen via URL.
* **Capability-Based Security (OCAP):** Zugriff auf Systemressourcen erfordert fälschungssichere Capability-Tokens als Parameter.
* **Hibernation-Daemons:** Einfrieren von Agenten bei VRAM-Mangel (Snapshots im Vektorgraphen) und reibungslose Rehydrierung.

Der 5-Phasen-Plan für den KI-Agenten

Ein LLM kann keinen kompletten Compiler in einem Durchgang schreiben. Du bist der Lead-Architekt, das LLM ist dein Junior-Entwickler. Nutze Divide & Conquer und zwinge das LLM zu Test-Driven Development (TDD).

Phase 1: Setup & Der Lexer

Ziel: Code in Tokens zerlegen.
Auftrag: "Nutze das Crate logos. Definiere ein enum Token für Keywords (agent, contract). Schreibe Tests."
Stoppe hier, bis alle Tests grün sind!

Phase 2: Der AST (Datenstrukturen)

Ziel: Die Architektur der Sprache im Speicher.
Auftrag: "Erstelle die Rust-Strukturen für den Abstract Syntax Tree (z.B. AstNode::AgentDef). Schreibe keine Logik, nur Datenstrukturen."

Phase 3: Der Parser

Ziel: Von Tokens zum Baum.
Auftrag: "Schreibe einen handgeschriebenen Recursive-Descent-Parser. Nimm Funktion für Funktion. Beginne nur mit Variablen-Deklarationen. Schreibe Tests dafür."

Phase 4: Semantische Analyse

Ziel: Typen-Check & OCAP-Sicherheit.
Auftrag: "Schreibe einen TypeChecker. Prüfe strikt, dass Funktionen, die Tools nutzen, Capability-Tokens haben."

Phase 5: Rust Code Generation

Ziel: Nativen Maschinencode über Rust-Transpilation erzeugen.
Auftrag: "Übersetze den typgeprüften AST in validen Rust-Quellcode. Agents → Structs, Contracts → Traits. Kompiliere via cargo build --release."

System-Prompts für den Coding-Agenten

Füttere deinen Agenten strikt nacheinander mit diesen Prompts. Gehe erst zur nächsten Phase, wenn der Code kompiliert und alle Tests grün sind.

Phase 1: Projekt-Setup & Lexer

Du bist ein Senior Compiler Engineer (Experte in Rust). Wir entwickeln "Varg" (.varg), eine KI-native Programmiersprache mit C#-ähnlicher Syntax. Wir arbeiten STRENG nach Test-Driven Development (TDD).

PHASE 1: Lexer
1. Initialisiere ein neues Rust-Projekt. Füge das Crate `logos` als Abhängigkeit hinzu.
2. Erstelle eine Datei `lexer.rs`. Definiere dort ein `enum Token` mit `#[derive(Logos)]`.
3. Benötigte Tokens: Keywords (`contract`, `agent`, `struct`, `public`, `private`, `import`, `implements`), KI-Typen (`Prompt`, `Context`, `Tensor`), Symbole ( `{`, `}`, `(`, `)`, `;`, `:` ), Identifier und String-Literale.
4. SCHREIBE ZUERST UNITTESTS: Teste mindestens einen einfachen Contract-Block und einen Agent-Block als reinen String, der in Tokens übersetzt wird.
5. Implementiere erst dann die Regex-Muster im Token-Enum, bis alle Lexer-Tests grün sind.
Schreibe noch KEINEN Parser und KEINEN AST! Fokus nur auf den Lexer.

Phase 2: Abstract Syntax Tree (AST)

Hervorragend, der Lexer funktioniert. Wir gehen über zu Phase 2.

PHASE 2: Abstract Syntax Tree (AST)
1. Erstelle eine Datei `ast.rs`. 
2. Definiere die Datenstrukturen (Structs und Enums), um die Varg-Sprache im Speicher zu repräsentieren.
3. Wichtige Knoten (Nodes): 
   - `Program` (Wurzelknoten, enthält eine Liste von Statements)
   - `ContractDef` (Name und Methoden-Signaturen)
   - `AgentDef` (Name, implementierte Contracts, Felder, Methoden)
   - `MethodDecl` (Sichtbarkeit, Rückgabetyp, Name, Parameter)
   - `Type` (Eingebaute wie Prompt/Context und Custom Types)
4. Schreibe KEINE Parsing-Logik! Es geht hier rein um die saubere Datenmodellierung in Rust. 
5. Schreibe einen Dummy-Test, der manuell (per Hand im Code) einen kleinen `AstNode::AgentDef` Baum zusammenbaut, um zu prüfen, ob die Typen und Lebensdauern (Lifetimes/Box) in Rust Sinn ergeben.

Phase 3: Der Parser

Der AST steht. Jetzt verbinden wir Lexer und AST durch den Parser.

PHASE 3: Recursive Descent Parser
1. Erstelle eine Datei `parser.rs`. Der Parser nimmt einen Iterator von unseren Logos-Tokens und produziert AST-Knoten. Setze auf Fehlerbehandlung mit `Result`.
2. Gehe iterativ vor (TDD): 
   - Schreibe einen Test für das Parsen einer einfachen Typ-Deklaration (`Prompt p;`). Implementiere die Methode `parse_declaration`.
   - Schreibe einen Test für einen `contract` Block. Implementiere `parse_contract`.
   - Schreibe einen Test für einen `agent` Block. Implementiere `parse_agent`.
3. Achte streng darauf, dass geschweifte Klammern korrekt gematcht werden. Wenn ein unerwartetes Token kommt, wirf einen sauberen, lesbaren `ParseError` (ähnlich wie in Rusts `miette` oder `codespan`).
Mache erst weiter, wenn der Parser einen echten `.varg` String fehlerfrei in deinen AST verwandelt.

Phase 4: Semantische Analyse (Type Checker & OCAP)

Wir haben jetzt einen AST. Bevor wir kompilieren, müssen wir den Code semantisch validieren.

PHASE 4: Type Checker & Capability Validation
1. Erstelle eine Datei `type_checker.rs`. Schreibe einen Visitor, der den gesamten AST durchläuft.
2. Prüfe auf Basis-Typfehler (z.B. Rückgabewert stimmt nicht mit Signatur überein).
3. Implementiere die OCAP-Sicherheitsregel: Wenn ein Agent eine Funktion aufruft, die ein Capability-Token erfordert (z.B. `NetworkAccess`), prüfe, ob dieses Token in der Parameterliste der Aufrufer-Methode existiert und weitergereicht wird.
4. TDD: Schreibe Tests mit absichtlich FALSCHEM Code (z.B. ein Agent versucht Netzwerk-Funktionen ohne `NetworkAccess` Parameter aufzurufen). Der TypeChecker MUSS hier einen Fehler werfen.

Phase 5: Codegenerierung (Rust Transpilation)

Der Code ist geparst und validiert. Jetzt erzeugen wir nativen Maschinencode via Rust-Transpilation.

PHASE 5: Rust Code Generation
1. Erstelle `codegen.rs` im Crate `varg-codegen`.
2. Implementiere einen `RustGenerator`, der den gesamten AST durchläuft.
3. Übersetze die AST-Knoten in validen Rust-Code:
   - `agent` → Rust `struct` + `impl` Block.
   - `contract` → Rust `trait`.
   - Varg-Methoden → Rust Funktionen mit korrekten Signaturen.
   - Runtime-Helfer (Networking, Crypto, LLM, DB) werden als Funktionen injiziert.
4. Lege den generierten Code als Cargo-Projekt in `.vargc_cache/` ab.
5. Kompiliere via `cargo build --release` zu nativen Binaries.
6. Schreibe einen Test, der einen einfachen Varg-Agenten als String einliest, ihn parst, typ-checkt und am Ende den generierten Rust-Code validiert.

Varg (.varg) Syntax-Beispiel

import "github.com/ai-os/stdlib/network";
import "github.com/ai-os/stdlib/system";

public contract WebSearcher {
    Result<List<String>, Error> Search(String query);
}

public agent ResearchAgent implements WebSearcher {
    
    private Context _memory;

    public ResearchAgent() {
        this._memory = new Context(WindowMode.AutoEject);
    }

    public Result<List<String>, Error> Search(String query, NetworkAccess netToken) {
        
        Prompt p = new Prompt($"Finde die neuesten Artikel über: {query}");
        Result<String, Error> response = LLM.Generate(p, this._memory);
        
        if (response.IsError) {
            return Result.Fail(response.Error);
        }

        return netToken.ExecuteSearch(response.Value);
    }
}

public static void Main(SystemCapability sys) {
    NetworkAccess netToken = sys.RequestNetwork();
    agent researcher = OS.SpawnDaemon(new ResearchAgent());
    
    researcher.Search("AI Operating Systems", netToken);
}
